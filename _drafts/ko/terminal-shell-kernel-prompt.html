사실 이 네 단어는 오늘 처음 만난 게 아니었어요. 1편에서 CLI와 GUI의 실행 흐름을 그려볼 때, 마지막 단계를 그냥 "둘 다 커널에 요청을 보낸다"라고만 적고 넘어갔었거든요. 그때는 "커널"이 뭔지 제대로 몰라서 뭉뚱그린 거였어요. 오늘 5교시가 정확히 그 뭉뚱그린 부분 — 터미널, 셸, 커널, 프롬프트가 각각 뭐고 어떻게 이어지는지 — 를 다루는 시간이었어요. 그래서 오늘 새로 배운 내용에, 1편을 쓸 때 AI와 나눴던 이야기까지 합쳐서 정리해볼게요.

이름의 역사부터 알아보기

수업 자료에 따르면 이 네 단어는 컴퓨터가 생기기 전부터 있던 말이 컴퓨터 용어로 옮겨온 경우예요.

  • 터미널(terminal)은 원래 "끝, 말단"이라는 뜻이에요. 통신·전기 분야에서 선로의 끝단을 가리키던 말이었어요. 1960~70년대에는 값비싼 대형 컴퓨터 한 대를 여러 사람이 나눠 썼는데, 사람들은 본체 앞이 아니라 키보드와 화면이 달린 "말단 장치"로 중앙 컴퓨터에 연결했어요. 그 물리 장치가 터미널이었고, 지금 우리가 쓰는 Windows Terminal이나 macOS Terminal은 그 옛 장치를 화면 속에서 흉내 내는 "터미널 에뮬레이터"예요.
  • 셸(shell)은 "껍데기"예요. 운영체제의 핵심(커널)을 감싸는 바깥층이라는 비유에서 온 이름이에요. 달걀을 떠올리면 딱 맞아요 — 껍데기가 알맹이를 감싸고 있죠.
  • 커널(kernel)은 "알맹이, 핵심"이에요. CPU·메모리·파일·네트워크 같은 컴퓨터의 진짜 자원을 관리하는 운영체제의 중심부예요.
  • 프롬프트(prompt)는 "재촉하다, 응답을 요구하다"라는 뜻이에요. 셸이 "나 지금 입력 기다리고 있어요"라고 화면에 띄우는 표시가 프롬프트예요.

이 중에서 특히 프롬프트의 어원이 재미있었어요. 4편에서 다뤘던 "AI 코딩 프롬프트"도 결국 같은 단어, 같은 뜻이거든요 — AI에게 "응답을 재촉하는" 문장이 AI 프롬프트이고, 셸이 사람에게 "입력을 재촉하는" 표시가 셸 프롬프트예요. 재촉하는 주체와 방향이 반대일 뿐, "누군가의 응답을 기다리며 신호를 보낸다"는 원래 뜻은 똑같이 살아 있었던 거예요.

세션(session)이라는 말도 자주 나와요

터미널이나 AI 도구를 쓰다 보면 세션(session)이라는 말도 자주 마주쳐요. 세션은 원래 "회의, 수업 시간"처럼 시작부터 끝까지 이어지는 하나의 기간을 뜻하는 말이에요. 학교에서 "1교시 세션"이라고 하면, 그 시간이 시작해서 끝날 때까지 전체를 가리키는 것과 같아요.

컴퓨터에서도 똑같은 뜻으로 써요. 터미널 창을 열어서 명령어를 치다가 그 창을 닫으면, 그 사이의 전체 기간이 하나의 셸 세션이에요. Claude Code를 실행해서 대화를 시작했다가 끝내는 것도 하나의 AI 세션이고요. 웹사이트에 로그인해서 로그아웃할 때까지도 로그인 세션이라고 불러요. 세션이 끝나면 그 안에서 기억하고 있던 정보(예: 지금 위치, 로그인 상태)도 대부분 함께 사라져요 — 창을 닫았다가 새로 열면 pwd가 다시 처음 위치로 돌아가 있는 것도, 이전 세션이 끝나고 새 세션이 시작됐기 때문이에요.

명령이 지나가는 길 (Windows 기준)

수업 자료의 원래 그림은 macOS·Linux 기준이라 PTY, zsh·bash가 나와요. 저는 Windows를 쓰니까, 같은 원리를 제 환경에 맞는 이름으로 다시 그려봤어요 — 수업 자료에는 없는 부분이라, 이 대응 관계는 AI와 함께 직접 정리한 거예요.

정리하면, 터미널은 창(입출력 담당), 셸은 통역사(명령 해석과 실행 담당), 커널은 진짜 자원을 쥐고 있는 관리자예요. 1편에서 뭉뚱그렸던 "커널에 요청을 보낸다"는 사실 이 세 층을 다 거친 다음에야 일어나는 마지막 단계였던 거죠.

명령 한 줄이 실제로 거치는 단계

PowerShell에서 Get-Location(현재 위치 확인, pwd라는 별명으로도 씀)과 node -v(Node.js 버전 확인)를 각각 입력했을 때 어떤 차이가 있는지 따라가 봤어요. 이 둘을 고른 이유는, 하나는 PowerShell 안에서 바로 처리되는 "내장 명령"이고 다른 하나는 2편에서 다뤘던 "PowerShell 밖의 별도 프로그램"이라서예요.

  1. 터미널이 키 입력을 받는다. Windows Terminal이 글자를 화면에 보여주면서 ConPTY로 전달해요.
  2. ConPTY가 PowerShell에 글자를 넘긴다. Enter까지 모인 문장이 PowerShell 프로세스로 전달돼요.
  3. PowerShell이 문장을 해석한다. 명령 이름, 파이프(|), 옵션 같은 문법을 나눠서 읽어요.
  4. PowerShell이 명령을 어디서 찾을지 판단한다. Get-Location은 PowerShell에 이미 내장된 cmdlet이라 디스크를 뒤질 필요가 없어요. node는 내장되어 있지 않아서, $env:Path에 적힌 폴더들을 순서대로 뒤져서 node.exe를 찾아요.
  5. 필요하면 새 프로세스를 만든다. Get-Location은 PowerShell 프로세스 안에서 곧바로 처리돼서 새 프로세스가 생기지 않아요. 반면 node.exe는 완전히 별도의 새 프로세스로 실행돼요(Windows는 이 과정에 CreateProcess라는 기능을 써요 — macOS·Linux의 fork·exec와 비슷한 역할이에요).
  6. 프로그램이 시스템 콜로 커널에 요청한다. 파일을 읽거나 현재 경로를 확인하는 일은 반드시 커널에 요청해야 해요. 커널이 권한을 확인하고 자원을 내어줘요.
  7. 결과가 되돌아와 화면에 보인다. 표준 출력(stdout)에 쓰인 결과가 ConPTY를 거쳐 터미널 화면에 그려지고, 그 다음 PowerShell이 새 프롬프트를 띄워요.

2편에서 "Node.js와 npm은 대체 언제 실행되는가"를 다뤘을 때 그렸던 그림이 바로 이 ④~⑤단계의 확대판이었던 거예요. 그때는 "npm이 실행된다"라고만 말했는데, 오늘 보니 그 실행이라는 게 정확히는 "셸이 PATH에서 찾아서 새 프로세스로 띄우는 것"이었네요.

흔한 오해 바로잡기

오해 1 — "터미널 = 검은 창 = 명령어"는 사실이 아니에요. 색은 그냥 꾸미기일 뿐이라, 밝은 테마의 Windows Terminal도 터미널이고 VS Code 아래쪽 패널도 터미널이에요. 터미널(창) · 셸(통역사) · 명령(실제로 실행되는 것)은 서로 다른 세 가지예요.

오해 2 — "터미널과 셸은 하나로 묶여 있다"도 아니에요. 같은 Windows Terminal 창 안에서도 탭마다 PowerShell을 띄우거나, cmd.exe를 띄우거나, WSL의 bash를 띄울 수 있어요. 터미널은 그저 "그림을 그려주는 창"이고, 그 안에서 어떤 셸이 돌아가는지는 완전히 다른 선택이에요.

오해 3 — "프롬프트 기호도 입력해야 한다"는 것도 흔한 착각이에요. 문서에 PS C:\Users\user> node -v라고 적혀 있다면, PS C:\Users\user>는 지금 PowerShell이 보여주는 프롬프트일 뿐이고 실제로 입력할 것은 node -v뿐이에요.

실습 관찰 — 내 PowerShell의 각 층 확인하기

아래 명령은 컴퓨터를 바꾸지 않고 정보만 보여줘요. Windows PowerShell에서 한 줄씩 실행해보세요.

$PSVersionTable
Get-Command node
$env:Path
  • $PSVersionTable은 지금 내가 쓰는 셸이 정말 PowerShell인지, 몇 버전인지 보여줘요.
  • Get-Command node는 PowerShell이 node라는 명령을 어디서 찾았는지 알려줘요. 결과에 node.exe의 실제 경로가 나오면 정상이고, 못 찾으면 node가 $env:Path에 없다는 뜻이에요.
  • $env:Path는 PowerShell이 외부 명령을 찾을 때 순서대로 뒤지는 폴더 목록이에요. 세미콜론(;)으로 구분된 여러 경로가 나와요.

이 관찰, 저는 이미 한 번 겪어본 적이 있어요. 2편에서 Node.js를 설치했는데도 새로 연 창에서 "찾을 수 없다"는 에러가 났던 적이 있었잖아요. 그게 바로 이 $env:Path 목록이 아직 새로고침되지 않은 상태였던 거예요. 설치는 됐는데 지금 이 PowerShell 창이 들고 있는 $env:Path 목록에는 아직 반영이 안 된 상태였던 거죠. 오늘 이 명령으로 직접 그 목록을 눈으로 보니, 그때 왜 창을 새로 열어야 했는지 훨씬 분명해졌어요.

관찰 질문: $env:Path에 Node.js나 npm 관련 경로가 보이나요? Get-Command node의 결과가 실제로 존재하는 파일을 가리키나요? 답이 예상과 다르더라도 괜찮아요 — 그 다른 결과 자체가 AI에게 물어볼 좋은 재료가 돼요.

(참고) macOS·Linux 사용자는 echo $SHELL, which node, echo $PATH로 같은 걸 관찰할 수 있어요.

Claude Code는 이 중 어디에 있을까

Claude Code는 터미널도, 셸도, 커널도 아니에요. 셸 위에서 실행되는 프로그램이에요. PowerShell 창에서 claude를 입력하면, PowerShell이 $env:Path에서 claude를 찾아 새 프로세스로 실행해요. 그 안에서 Claude Code가 파일을 읽고, 명령 실행을 요청하고, 결과를 다시 보여주는 거예요.

이 구분을 알면 에러가 났을 때 어디를 봐야 할지 감이 잡혀요.

  • claude라는 이름을 인식할 수 없다는 에러 → 설치가 안 됐거나 $env:Path에 없는 경우 (셸의 명령 탐색 문제)
  • "액세스가 거부되었습니다" 같은 권한 에러 → 커널이 관리하는 파일·폴더 권한 문제
  • 글자가 깨지거나 창이 멈춤 → 터미널 앱이나 ConPTY 연결 자체의 문제

AI가 아무리 강력해도 지금 서 있는 폴더와 권한을 넘어서지는 못해요. 그래서 에러 문구, Get-Location 결과, 실행한 명령을 함께 보여주는 습관이 (2편에서 이미 이야기했던) "환경 / 명령 / 결과"를 나눠 보여주는 것과 똑같이, AI에게 더 정확한 도움을 받는 방법이에요.

참고한 자료: 터미널·셸·커널·프롬프트의 이름과 역사, PTY 개념, 명령이 거치는 단계는 오늘 수업 자료(Linux kernel 공식 문서, GNU Bash Manual, zsh 공식 문서, POSIX/MDN의 시스템 콜 설명, Unix 셸 역사 관련 자료를 인용)를 바탕으로 정리했어요. 여기에 macOS·Linux 개념을 Windows(ConPTY, PowerShell, CreateProcess)에 대응시킨 부분은 수업 자료에는 없는 내용이라, AI와 함께 직접 정리해본 것이에요.
