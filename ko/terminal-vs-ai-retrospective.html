← 이전 편 Claude Code로 파일 정리 자동화하기 (dry-run 프롬프트) — 바이브 코딩으로 웹 만들기 (7)

수업 자료에는 "두 방식 중 하나가 항상 낫지는 않다"고 적혀 있었어요. 직접 명령은 지금 내 위치와 파일 하나하나를 세밀하게 이해하게 해주고, AI 협업은 반복적인 목록화와 계획 작성 속도를 높여준다고요. 그리고 어느 쪽이든 실행 결과를 검토하는 책임은 나에게 있다는 것도요. 6편(손으로 직접)과 7편(Claude Code에게 맡김)을 실제로 둘 다 해봤으니, 이번엔 그 둘을 나란히 놓고 비교해봤어요.

직접 한 것과 AI에게 맡긴 것, 나란히 놓고 보기

비교 질문직접 한 것 (6편)AI에게 맡긴 것 (7편)
무엇을 어떻게 옮길지 mkdir, mv를 한 줄씩 제가 직접 결정했어요. Claude Code가 dry-run으로 "현재 위치 → 제안 위치" 목록을 먼저 제안하고, 제가 승인했어요.
내가 확인한 증거 pwd, ls 출력을 직접 눈으로 확인했어요. AI의 dry-run 목록을 먼저 읽고, 승인한 뒤에는 다시 ls로 재확인했어요.
가장 불안했던 지점 3단계에서 cd를 여러 번 오가다가 지금 위치가 헷갈렸던 순간. draft.docx처럼 애매한 파일을 AI가 마음대로 분류해버릴까 걱정했던 순간.
다음번 더 명확히 할 규칙 폴더를 옮기기 전엔 항상 pwd로 위치를 먼저 재확인하기. 애매한 파일은 처음부터 "질문 목록으로 따로 받기"를 프롬프트 규칙에 못박아두기.

표로 나란히 놓고 보니 확실해진 게 있어요. 직접 할 때 불안한 지점은 "내 위치를 잃어버리는 것"이었고, AI에게 맡길 때 불안한 지점은 "AI가 애매한 걸 마음대로 정해버리는 것"이었어요. 둘 다 결국 "판단 근거를 눈으로 직접 보는가"라는 같은 문제의 다른 얼굴이었던 거예요.

오늘 하루를 세 문장으로 정리하기

수업 자료가 마지막으로 남긴 세 가지 질문에, 오늘 실제로 겪은 내용으로 답해봤어요.

1. 오늘 pwd가 알려준 "내 위치"는 어디였나요?

3편에서 만든 my-first-web 폴더와 같은 자리, Documents 폴더 밑에 terminal-practice라는 이름으로요. 정확한 경로 자체보다, pwd를 실행할 때마다 "지금 여기가 어디인지"를 추측이 아니라 확인으로 알 수 있다는 확신이 생긴 게 더 큰 소득이었어요.

2. AI의 dry-run에서 내가 멈추거나 수정한 제안은 무엇이며, 왜였나요?

7편에서 draft.docx를 분류하지 말고 그대로 두라고 했던 게 그거예요. AI가 확장자만 보고 "문서니까 여기"라고 넘겨버릴 수도 있었는데, 초안인지 완성본인지 저도 확신이 없는 상태였거든요. 확실하지 않은 걸 AI가 대신 정해버리게 두면, 나중에 그 파일을 찾을 때 제가 스스로도 왜 거기 있는지 설명할 수 없게 돼요. 그래서 "모르는 건 미뤄두기"를 선택했어요.

3. 다음 프로젝트에서 Node.js 버전을 바꿔야 한다면, nvm을 어떻게 쓸까요?

2편에서 다뤘던 내용을 그대로 가져오면, 윈도우에서는 nvm install에 버전 번호를 직접 적어야 해요(윈도우용 nvm은 --lts 같은 줄임 옵션이 없다고 했었죠). 그러니 저는 먼저 nvm-windows 공식 문서에서 그 시점의 최신 LTS 버전 번호를 확인하고, nvm install <버전 번호>로 설치한 다음 nvm use <버전 번호>로 전환할 것 같아요.

이번 8편을 끝으로, "바이브 코딩으로 웹 만들기" 시리즈를 마칩니다. 1편에서 "작업대"가 왜 필요한지도 몰랐던 게 오늘의 시작이었는데, 8교시를 지나면서 그 작업대 위에서 직접 손을 움직여보기도 하고, AI에게 맡겨보기도 하고, 그 둘을 비교까지 해봤어요. 다음에 또 새로운 수업이 생기면 그때 새 시리즈로 이어서 정리할게요.

참고한 자료: 비교 회고의 질문 항목(비교 질문 표, 마무리 세 문장)은 오늘 수업 자료를 바탕으로 정리했어요. 표와 질문에 채운 답은 이번 시리즈(1~7편)에서 실제로 겪고 정리했던 내용을 가져온 것이에요.
