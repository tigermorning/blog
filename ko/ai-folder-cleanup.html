6편 끝에 남겨둔 예고 그대로, 이번엔 6편에서 만든 terminal-practice 폴더 정리를 Claude Code에게 맡겨봤어요. 그런데 목표가 "AI가 대신 치워주는 것"이 아니었어요. 분류 계획을 먼저 내고, 제가 승인한 뒤에만 실제로 옮기게 하는 것이 이번 실습의 핵심이었어요. 이건 4편에서 다룬 "지시 → 생성 → 판단 → 재지시" 루프를 그대로 쓰는 거예요. 다만 이번엔 되돌리기 어려운 파일 이동이 걸려 있어서, "판단" 단계를 실행 전으로 앞당긴 거예요.

먼저 폴더를 좀 더 어지럽혀 봤어요

6편에서 만든 terminal-practice 폴더에는 ideas, assets/images/landing-page.txt, archive가 있었죠. 여기에 연습 삼아 screenshot.png, notes.txt, draft.docx 세 개를 폴더 맨 위(어디에도 안 들어간 상태)에 그냥 흩어뒀어요. 실제 바탕화면이 아니라 이 연습 폴더에서만 하는 거라, 마음 편하게 어지럽힐 수 있었어요.

Claude Code에게 보낸 프롬프트

수업 자료에는 정리를 맡길 때 지켜야 할 규칙이 개조식으로 나와 있었어요. 4편에서 다뤘던 "추가할 것 / 지킬 것"과 같은 구조예요. 저도 같은 형태로 요청했어요.

terminal-practice 폴더의 파일과 폴더를 읽고 정리 계획을 세워줘.
지킬 것:
- 삭제 명령(rm 등)은 절대 실행하지 마.
- 먼저 dry-run으로 "현재 위치 → 제안 위치" 목록과 분류 기준만 보여줘.
- 이동은 내가 목록을 검토하고 "승인"이라고 답한 뒤에만 실행해.
- 이름을 추측하지 말고, 확장자와 현재 이름을 근거로 분류해.
- 실행 뒤에는 이동한 목록과 남은 파일 목록을 보여줘.

dry-run이 뭐길래 이렇게 강조할까

dry-run은 실제로 아무것도 바꾸지 않고 "이대로 실행하면 무슨 일이 일어날지"만 미리 보여주는 걸 말해요. 이번 수업에서 새로 나온 개념이었어요. 4편에서 "판단" 단계는 결과가 나온 뒤에 검토하는 거였는데, dry-run은 그 판단 시점을 결과가 나오기 전으로 옮겨요. 파일을 실제로 옮기고 나서 잘못을 알게 되면 되돌리기 번거롭잖아요. 그래서 AI가 먼저 mv를 실행하려고 하면, "아직 실행하지 말고 dry-run 목록만 보여줘"라고 바로 멈춰야 해요.

제가 받은 dry-run 결과는 이런 식이었어요.

현재 위치제안 위치분류 기준
screenshot.pngassets/images/screenshot.png이미지 확장자(.png)
notes.txtideas/notes.txt텍스트 메모로 추정
draft.docx(분류 보류, 질문)초안 문서인지 완료본인지 확실치 않음

draft.docx를 AI가 마음대로 넘기지 않고 "질문"으로 남긴 부분이 특히 마음에 들었어요. 이름을 추측하지 말라는 규칙이 실제로 지켜진 증거였거든요.

AI 산출물 검토 체크리스트

결과를 그대로 승인하기 전에, 다섯 가지를 확인했어요.

확인할 것통과 기준내 판단
파일 이동 전 dry-run각 파일의 현재 위치와 제안 위치가 목록으로 있다통과
삭제 방지삭제 명령·삭제 제안이 없고, 보류 파일은 그대로 둔다통과
분류 기준확장자·용도 등 읽을 수 있는 기준이 있고, 애매한 파일은 질문한다통과
경로 안전작업 폴더 밖의 파일을 건드리지 않고, 폴더를 통째로 옮기지 않는다통과
실행 후 확인ls 또는 목록으로 이동 결과와 남은 항목을 보여준다실행 후 확인 예정

승인하고, 한 가지는 수정 요청하기

계획이 대체로 마음에 들었지만, draft.docx는 지금 당장 정하고 싶지 않았어요. 그래서 이렇게 답했어요.

계획을 검토했어. screenshot.png와 notes.txt는 제안한 대로 옮겨줘.
draft.docx는 지금 분류하지 말고 그대로 폴더 맨 위에 둬.
삭제·이름 변경은 하지 마. 실행 후 이동 전후 목록을 비교해서 보여줘.

이 문장도 4편에서 배운 개조식 그대로예요 — "무엇은 진행, 무엇은 보류, 무엇은 하지 마"를 한 줄씩 나눠서 답하니 AI가 헷갈릴 여지가 줄었어요. 실행 후 ls로 확인해보니 screenshot.png와 notes.txt만 옮겨졌고, draft.docx는 그대로 남아 있었어요 — 승인한 그대로 딱 맞게 실행된 거예요.

손으로 직접 mv를 쳤던 6편과 비교하면, 이번엔 계획을 세우는 속도가 훨씬 빨랐어요. 대신 계획이 맞는지 확인하는 책임은 여전히 저에게 있었고요. 이 둘을 나란히 놓고 비교해보는 게 다음 8편이에요.

참고한 자료: 실습 목표, 프롬프트 규칙, dry-run 개념, 검토 체크리스트의 항목은 오늘 수업 자료를 바탕으로 정리했어요. 실제로 어지럽힌 폴더 상태와 AI가 낸 dry-run 결과, 제 승인·수정 요청 문장은 이번에 직접 실습해본 내용이에요.
